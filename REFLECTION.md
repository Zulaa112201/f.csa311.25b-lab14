
REFLECTION.md
1. Аль assertion (тест) хамгийн их үнэ цэнэтэй санагдсан вэ? Яагаад?
Энэхүү лабораторийн ажлын хүрээнд нийт 5 гаруй төрлийн assertion бичиж үзсэнээс миний хувьд JSON Schema / Property validation болон Business rule шалгалтууд хамгийн их үнэ цэнэтэй бөгөөд ач холбогдолтой санагдсан. Зөвхөн HTTP статус код 200 буцаж байгааг шалгах нь тухайн API-ийг бүрэн зөв ажиллаж байна гэж дүгнэхэд хангалтгүй юм. Серверээс амжилттай хариу ирсэн ч дата дутуу, бүтэц нь өөрчлөгдсөн, эсвэл буруу утгатай ирэх эрсдэл үргэлж байдаг.  


JSON Schema-г шалгаснаар API-ийн гэрээ (Contract) зөрчигдсөн эсэхийг шууд илрүүлэх боломжтой болдог. Жишээ нь, хэрэглэгчийн мэдээллийг авах үед id нь заавал number байх, email талбар нь заавал байх ёстой гэдгийг баталгаажуулснаар  урд талын систем (Frontend) дээр үүсэж болох гэнэтийн undefined болон төрлийн зөрүүтэй (Type mismatch) алдаануудаас урьдчилан сэргийлдэг. Мөн бизнесийн дүрэм (Жишээ нь: и-мэйл хаяг заавал @ тэмдэгттэй, regex форматад нийцсэн байх) шалгах нь  системийн логик зөв ажиллаж байгааг баталгаажуулах хамгийн чухал хамгаалалтын шугам болдог байна.  


2. Negative test 1+ жишээгээ дэлгэрэнгүй тайлбарла, яг ямар алдааг олох вэ?
Системийн зөвхөн "хэвийн ажиллах" (Happy path) замыг шалгах нь хангалтгүй бөгөөд алдааны урсгал буюу сөрөг тестийг заавал хийх шаардлагатайг ойлгосон. Миний бичсэн сөрөг тестийн нэг жишээ бол системд бүртгэлгүй буюу оршин байхгүй хэрэглэгчийн ID-аар (GET /users/999999) мэдээлэл дуудах кейс юм.  

Энэхүү сөрөг тестийн гол зорилго нь сервер байхгүй өгөгдөлтэй тулгарахдаа системийг бүхэлд нь унагахгүйгээр, үйлчлүүлэгчид ойлгомжтой 404 Not Found статус код болон стандартчилагдсан алдааны мессеж буцааж чадаж байгааг шалгах юм. Хэрэв энэ тест унаж, сервер 500 Internal Server Error буцаавал эх кодод алдаа барих (Exception handling) хэсэг дутуу, эсвэл бааз руу хийсэн query нь амжилтгүй болсноос үүдэн сервер дотооддоо гацсаныг илрүүлж чадна. Мөн POST request дээр дутуу эсвэл форматын алдаатай payload явуулж 400 Bad Request ирж байгааг шалгах сөрөг тест нь  серверийн input validation (оролтын өгөгдлийн шалгалт) зөв ажиллаж буйг баталгаажуулдаг.  


3. Postman дотор амжилттай ажиллаж байсан тест Newman-д орвол fail болсон уу? Яагаад?
Postman GUI орчинд Send товч дарахад амжилттай Pass болж байсан тестүүд Newman CLI буюу командын мөрнөөс ажиллуулах үед Fail болох тохиолдол гарсан. Үүний хамгийн гол шалтгаан нь Орчны хувьсагч (Environment Variables) болон Төлөвийн хамаарал (State Dependency) байв.  


Postman дээр ажиллах үед өмнөх request-үүдээс хадгалагдсан глобал эсвэл active environment-ийн хувьсагчид (жишээ нь хуучин dynamic token) санах ойд нь хадгалагдаад үлдчихдэг. Гэтэл Newman дээр ажиллуулахдаа export хийсэн env.dev.json файлыг шинээр уншуулдаг тул хэрэв бид request хоорондын гинжин хэлхээг (Request chaining) зөв тохируулаагүй, эсвэл token-ыг автоматаар шинэчилж авдаггүй бол тестүүд шууд 401 Unauthorized алдаа зааж унадаг. Мөн local-д ажиллаж байгаа серверийн baseUrl хаягийг Newman-д зөв дамжуулаагүй эсвэл asynchronous request-ийн хариу ирэх хугацаа сүлжээнээс хамааран Newman-ий timeout-оос хэтэрснээс болж унах тохиолдол гардаг. Энэ нь тестийг ямар ч тусгаарлагдсан орчинд бие даан, цэвэр ажилладаг (Stateless) байхаар загварчлах ёстойг ойлгуулсан.  


4. Token эсвэл secret-ыг хэрхэн зохицуулсан вэ? (env-ийн соёл)
Аюулгүй байдлын үүднээс private token, нууц үг, API key зэрэг эмзэг мэдээллийг хэзээ ч эх код болон Git repository руу шууд хадгалж (hardcode) commit хийж болохгүй гэдгийг хатуу анхаарсан. Үүний тулд "Environment-ийн соёл"-ыг дагаж ажилласан.  

Postman дээр хөгжүүлэлт хийхдээ baseUrl болон token-ыг орчны хувьсагч болгон зохион байгуулж , collection-оо экспортолж авахдаа env.dev.json болон env.ci.json файлууд дахь бодит нууц үг, token-ий утгыг устган, оронд нь REAL_TOKEN_HERE эсвэл YOUR_SECRET_TOKEN гэсэн placeholder утгуудыг үлдээсэн. Харин GitHub Actions СІ pipeline дээр Newman-ийг ажиллуулахдаа бодит нууц үгсийг задруулахгүйн тулд GitHub Secrets (${{ secrets.API_TOKEN }}) ашиглан , ажиллах явцад нь placeholder утгыг substitute хийх (орлуулах) замаар аюулгүй байдлыг бүрэн хангаж сурсан.  


5. API өөрчлөгдвөл collection-ийн аль хэсэг ХАМГИЙН ИХ эвдрэх вэ? Үүнийг тестийн "эмзэг газар" гэж нэрлэдэг, яаж бууруулах вэ?
API өөрчлөгдөх эсвэл шинэчлэгдэх үед collection-ийн хамгийн их эвдрэх "эмзэг газар" (Fragile point) бол Эхлэл цэгүүдийн URL (Endpoints) болон Response-ийн бүтцийн хатуу шалгалтууд (Hardcoded Payload/Schema assertions) байдаг. Хэрэв олон request дээр URL-ийг гараар хатуу бичсэн бол суурь хаяг өөрчлөгдөхөд бүгд унах аюултай.
